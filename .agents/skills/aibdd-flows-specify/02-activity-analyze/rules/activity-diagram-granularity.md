# Activity 圖（流程）顆粒度

- **計數單位**：對外產出之**活動圖數量**，以「**一條可獨立驗收之業務流程**」（一條 **independent flow**）為準——**至多一張活動圖對應一條 flow**。**不得**以 feature 檔案數、API 數、技術分層或模組邊界為由預設「一檔一圖」。
- **獨立流程**：須具備可對使用者或驗收方陳述之 **進場條件／觸發** 與 **可驗收之終局**（成功、失敗、或契約可述之中止）。僅列舉能力、規格項目或 feature 名稱而**無**明顯控制流主軸與狀態演進者，**不**另計為需獨立活動圖之流程。
- **Rework 迴圈**：**退回→補件→再送→再審／再評估** 等同一段業務旅程上之迭代，**不得**拆成多張活動圖。
- **同一 function package**：多個 Actor 若處於**同一條**連續因果鏈、後段狀態依賴前段，**通常**收斂為**單張**活動圖；**不得**僅因角色標籤不同而增加張數。
- **純查詢／唯讀瀏覽**：無業務狀態遷移、無獨立驗收里程碑者，**不**外拉獨立活動圖；敘述收斂於 feature／rule 即可。
- **合併 vs 拆開**：當敘述上存在**兩條**可各自講完、**入口或主線驗收點明顯不同**之對外業務主線，**得**對應**兩條** flow、因而**兩張**活動圖；若僅為同一主線下之變體強度（happy／極小／極大）或情境覆寫，**優先**以**同張**圖內分支／註記表達，**不**先膨脹張數。

# 反例

- 同一案件 **送出→遭退件→補件→再送→結案** 拆成兩張活動圖，把 rework **誤當**第二條獨立流程。
- 同包內 **審核員核准** 與 **主管覆核** 實為串接之同一條審批鏈，卻因身分標籤拆兩張圖。
- 僅因存在 **三份 `.feature` 檔** 就畫 **三張** 活動圖，實際上只有 **一條** 使用者旅程與 **一組** 連續狀態。預期應**一張**圖涵蓋該 journey，多檔 feature 仍綁在同一 flow。
- 把 **儀表板列表／查詢** 獨立成張活動圖，而主敘事無狀態遷移之獨立驗收。預期應不另立圖，或併入具寫入／狀態改變之主流程之讀取段若必須出現在敘事中。
