# Actor 顆粒度

- Activity 圖以 **UI-facing 人類或組織角色**為主軸。
- **不得**將產品內建系統、自有後端、排程、worker、資料庫列為獨立 Actor。
- **外部使用者**（透過 UI／CLI 等由人主動操作觸發者）得是 Actor；同一 Activity 內 **Actor 名稱唯一**，domain 同一角色不得重複多列。
- **第三方系統**僅當該段 flow 的**入口**為 Webhook、Callback、或邊界外服務主動打進來且為**觸發端**時，得為 Actor。
- Webhook／Callback 情境下，下游金流／寄信／一般外部 API **若發生在使用者已在 UI 觸發之後**，**不得**升格為 Actor；其結果歸入觸發該流程之 Action 的 boundary／postcondition／對應 feature 之規則敘述。

# 反例

- 將 **排程器、訊息佇列 consumer、資料庫、自有後端服務**畫成與人類並列的 Actor，彷彿它們「主動開局」一條業務流程。
- 使用者在 **UI 按下付款**後，另立 **金流閘道**、**寄信服務**為 Actor，與「已按付款的人」並列，把非入口的下游互動當成圖上獨立身分。
- 入口其實是 **使用者填表送出**，卻額外畫 **Webhook 接收端**當主 Actor（若該段並非由邊界外主動觸發開局）。
- 同一 Activity 內同時出現 **「玩家」與「使用者帳號」**（或兩個稱呼不同、domain 上同一操作者），違反名稱唯一／不重複多列。
