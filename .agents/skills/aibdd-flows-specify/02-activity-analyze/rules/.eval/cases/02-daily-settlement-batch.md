# Benchmark 02 — 每日結算批

## 埋藏陷阱

- 提交結算後系統**自動**產生結算批、撥款、進入已結算。
- 撥款失敗**四類原因**可獨立命名，易各拆一 feature。
- 呼叫金流、更新帳務屬**內部鏈**；「進入結算中」易成純狀態 API 型 feature。

## 需求原文

財務人員每日關帳：可先**上傳對帳檔**，系統比對後供財務**確認對帳結果**（確認無誤或標記差異已處理）。

當日所有對帳項目皆已確認後，財務可**提交結算申請**。提交後系統**自動**產生當日結算批、**自動**執行撥款；撥款完成即進入「已結算」狀態。

若撥款失敗，系統依失敗原因分四類處置：帳戶異常、額度不足、路由錯誤、需人工覆核。財務人員**不能**手動按「產生結算批」「執行撥款」「切換已結算」；亦無獨立 API 僅用於狀態切換。

內部會呼叫金流閘道、更新帳務狀態，皆為提交結算後之技術鏈，不對外暴露為使用者操作。

## 預期產出之 feature file

- `上傳對帳檔.feature`
- `確認對帳結果.feature`
- `提交結算申請.feature`

## 不應出現（常見誤拆）

- `產生結算批.feature`、`執行自動結算.feature`、`執行撥款.feature`、`進入結算中.feature`
- `撥款失敗帳戶異常.feature`、`撥款失敗額度不足.feature`、`撥款失敗路由錯誤.feature`、`撥款失敗人工覆核.feature`
- `呼叫金流閘道.feature`、`更新帳務狀態.feature`

四類失敗為 `提交結算申請` 內 **Scenario**；自動產批與撥款為其 **postcondition**。
