# Rule 疑慮分析切角

1. 目的
   1. 本檔只列出在 `01-atomic-rules-analyze`（`/aibdd-rules-specify`）中，對當前 `.feature` 內 `Rule:` 可能產生疑慮的分析切角。
   2. 本檔將切角分成兩類：
      1. `$NEED_TO_CLARIFY`：代表目前需求真相不足，後續應交給外層 clarify 流程。
      2. `$NEED_TO_FIX`：代表目前 rules 已可直接判定有問題，應回到前面步驟重推或重寫，不必先訪談。

## $NEED_TO_CLARIFY
1. 推理來源的模糊
   1. 要看什麼：此 `.feature` 綁定之 `Action` 是否真的能在 `${PLAN_SPEC}` 找到對應 user raw idea 段落。
   2. 需求訪談重點：要先確認這個 `.feature` 對應的業務動作是否真的存在於使用者需求裡，還是 analyst 把其他敘事誤投射成這個 action。
   3. 例：Feature 是「顧客提交訂單」，但需求全文只寫了「系統會顯示最新促銷活動」，沒有任何提交訂單的敘述。
2. 證據不足
   1. 要看什麼：雖然 user raw idea 有提到相關主題，但證據是否足以推出目前這條 rule 的完整主詞、條件與語氣。
   2. 需求訪談重點：要確認使用者真正想約束的欄位、條件、門檻與語氣，不要把模糊敘述硬收成精確規則。
   3. 例：user raw idea 只寫「系統要檢查輸入」，但 rule 寫成 `Rule: 前置（參數） - 訂單金額必須大於 0 且折扣碼必須符合活動期間`。
3. 規則彼此之間矛盾
   1. 要看什麼：同一 `.feature` 內不同 rule 是否互相否定、互相衝突，或同時要求不可能並存的結果。
   2. 需求訪談重點：要確認哪條規則代表真正需求，或是否其實少了前提條件、適用範圍或例外情境來解除衝突。
   3. 例：一條 rule 寫 `提交訂單前購物車商品數量必須大於 0`，另一條卻寫 `提交訂單前購物車必須為空`。
4. 規則疊加後缺少合成規則
   1. 要看什麼：多條 rule 個別成立，但一起作用在同一個計算、狀態轉移或決策點時，是否少了一條更高一層的規則來定義疊加順序、優先權或合成方式。
   2. 需求訪談重點：要確認多條規則同時命中時的優先順序、計算順序、互斥關係與最終合成結果。
   3. 例：訂單最後金額結算時，同時有折價券規則與折扣券規則，但 rules 沒有說兩者同時使用時，是先折價再折扣，還是先折扣再折價。
5. 類型前綴不確定
   1. 要看什麼：某條規則到底屬於 `前置（狀態）`、`前置（參數）`、`後置（回應）` 還是 `後置（狀態）`。
   2. 需求訪談重點：要確認使用者真正在意的是輸入限制、執行前狀態、對外輸出，還是執行後系統狀態。
   3. 例：`付款失敗要顯示錯誤訊息`，不確定是對外回應還是 UI/系統狀態改變。
6. 主詞或責任歸屬模糊
   1. 要看什麼：rule 的主詞到底是 actor、aggregate、system、external dependency 還是某個 domain object。
   2. 需求訪談重點：要確認是誰負責記錄、驗證、轉移狀態或對外回應，避免把不同責任主體混成同一條 rule。
   3. 例：`應記住上一次配送選項`，不清楚是購物車記住、顧客帳戶記住，還是 UI 暫存記住。
7. 同一 `.feature` 混入多個可獨立驗收的 operation
   1. 要看什麼：同一檔 rules 是否其實覆蓋了兩個以上可分開命名、分開觸發、分開驗收的 operation。
   2. 需求訪談重點：要確認這些 operation 在產品語意上是否應該拆成不同 feature，還是使用者本來就把它們視為單一能力。
   3. 例：同一檔同時有 `建立訂單` 與 `取消訂單` 的 rules。
8. 前置狀態缺口
   1. 要看什麼：user raw idea 明明暗示了必要前置狀態，但目前 rules 沒有把它顯式收進來。
   2. 需求訪談重點：要確認有哪些狀態是「沒滿足就不該觸發」的硬前提，以及這些前提是否要被明寫成規則。
   3. 例：需求寫「只有登入會員可以使用優惠券」，但 rules 沒有任何 `會員登入狀態必須成立` 類型的前置狀態規則。
9. 時序或狀態轉移不清
   1. 要看什麼：rule 所描述的是觸發前、觸發中、觸發後哪個時點，或狀態轉移前後關係是否混亂。
   2. 需求訪談重點：要確認條件與結果各自發生在什麼時點，哪些是 trigger 前提，哪些是 trigger 後保證。
   3. 例：`提交訂單後購物車必須已包含商品`，語意上把前置條件寫成後置時間點。

## $NEED_TO_FIX
1. 規則腦補
   1. 要看什麼：rule 是否自行長出 user raw idea 沒授權的數值、技術詞、隱性前提、介面詞或額外限制。
   2. 如何修正：回到步驟 `1.1`，刪掉所有未被 user raw idea 授權的內容，只保留能指回需求原文的主詞、條件與語氣，再重跑步驟 `1.2`。
   3. 例：user raw idea 只寫「失敗要通知顧客」，rule 卻寫成 `操作失敗應於 30 秒內以 email 通知顧客`。
2. 規則與 user raw idea 矛盾
   1. 要看什麼：rule 的主詞、條件、語氣或結果是否與需求原文相反、縮小、放大或扭曲。
   2. 如何修正：回到步驟 `1.1`，依 user raw idea 重新推理 rule wording，必要時先刪掉錯誤 rule，再把與需求一致的版本寫回步驟 `1.2`。
   3. 例：user raw idea 寫「必須先完成付款授權」，rule 卻寫成 `通常應完成付款授權`。
3. 規則彼此重複或語意重疊
   1. 要看什麼：兩條 rule 是否只是換句話說同一件事，或一條較窄、一條較寬但其實沒有新增獨立驗收價值。
   2. 如何修正：回到步驟 `1.1` 做去重，保留語意最完整、最直接對應需求原文的那一條，再於步驟 `1.2` 只寫回一條 rule。
   3. 例：`折扣碼必須為 8 碼字串` 與 `折扣碼長度必須等於 8` 同時存在。
4. 原子化不足
   1. 要看什麼：一條 rule 是否同時承載多主詞、多條件、多結果，或把前置與後置混在一起。
   2. 如何修正：回到步驟 `1.1` 依 `atomic-rule-granularity.md` 拆成多條單一主詞、單一條件、單一結果的 rule，再於步驟 `1.2` 寫回。
   3. 例：`Rule: 前置（狀態） - 購物車必須已建立且庫存必須充足且系統應建立訂單`。
5. 掛載到錯的 `Action`
   1. 要看什麼：某條 rule 是否其實屬於另一個 action，只是被錯掛到目前 `.feature`。
   2. 如何修正：回到步驟 `1.1` 重新比對本 phase 已綁定的 `Action` 與 source evidence，把 rule 重新分配到正確 action 所屬的 rule 集合，再重跑步驟 `1.2`。
   3. 例：`顧客加入商品到購物車後應更新購物車總額` 被寫進 `提交訂單.feature`。
6. 掛載到錯的 `.feature`
   1. 要看什麼：rule 雖然看起來合理，但它真正應該掛到別的 feature 檔，而不是現在這一檔。
   2. 如何修正：回到步驟 `1.1` 重新整理 feature 與 rule 的對應；若正確 `.feature` 已存在就移過去，若只是本輪寫回錯檔則直接在步驟 `1.2` 改寫正確檔案。
   3. 例：`查詢訂單列表應回傳最近十筆訂單` 被掛在 `提交訂單.feature`。
7. command 與 query 混檔
   1. 要看什麼：有副作用的規則與純查詢規則是否被放進同一 `.feature`。
   2. 如何修正：回到步驟 `1.1` 依 command/query 邊界重分 rule；若需要，將 query rules 改寫到對應查詢 feature，再重跑步驟 `1.2`。
   3. 例：同一檔同時有 `提交訂單後應扣庫存` 與 `查詢訂單歷史應回傳最近五筆`。
8. 參數規則缺口
   1. 要看什麼：此 `.feature` 是否完全缺少必要的 `前置（參數）` rule，或少了某個明顯必要的輸入約束。
   2. 如何修正：回到步驟 `1.1` 重新盤點 user raw idea 中已明講的輸入約束，補出缺失的 `前置（參數）` rule，再於步驟 `1.2` 寫回。
   3. 例：`建立訂單` 有大量狀態與回應規則，但沒有任何 `收件地址必須...` 或 `付款方式必須...` 類型的前置參數規則。
9. 結果面向缺口
   1. 要看什麼：user raw idea 已提到某個可驗收結果，但目前 rules 只覆蓋了其中一部分。
   2. 如何修正：回到步驟 `1.1` 把已被需求明示的結果面向補齊成獨立 rule，再於步驟 `1.2` 更新 `.feature`。
   3. 例：需求寫「付款成功時要建立訂單並發送確認通知」，rules 卻只有 `訂單應建立`，沒有 `應發送確認通知`。
10. 命名句型不合法
   1. 要看什麼：`Rule:` 是否符合 `Rule: <類型前綴> - <主詞> 必須／應 <單一條件>` 句型。
   2. 如何修正：回到步驟 `1.1` 依標準句型重寫 headline，確保類型前綴、主詞與 `必須／應` 動詞合法，再於步驟 `1.2` 覆寫舊句。
   3. 例：`Rule: 系統會驗證收件地址是否合法`，缺少合法類型前綴與 `必須／應` 動詞。
11. 規則語氣過弱或過強
   1. 要看什麼：rule 是否把 user raw idea 的強制語氣弱化，或把原本只是一般描述的內容強化成硬性約束。
   2. 如何修正：回到步驟 `1.1` 對齊 user raw idea 的語氣強度，避免把 `可` 改成 `必須`，或把 `必須` 改成 `通常`，再重跑步驟 `1.2`。
   3. 例：user raw idea 寫「可重新付款」，rule 卻寫成 `顧客付款失敗後必須重新付款`。
---
3. 使用提醒
   1. 同一條 rule 可以同時命中多個切角；不要因為它已被歸進某一類，就停止從其他角度檢查。
   2. 若同一個問題同時落在 `$NEED_TO_FIX` 與 `$NEED_TO_CLARIFY` 的邊界，先問自己：目前是「需求真相不足」還是「目前 rule 明顯寫錯」；前者歸 clarify，後者歸 fix。
