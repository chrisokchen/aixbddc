# Pre-Red Hook — nextjs-playwright wiring gate

## §0 Pre-Red 的職責邊界（NORMATIVE）

**Pre-Red 只建 wiring，不建 logic。**

通過 Pre-Red 後，下列**結構性事實**必須成立：

- `${CONTRACTS_DIR}/api.yml` 在 Plan phase 已寫定（本 phase 不撰寫 / 不修改 spec 本身）。
- 所有 contract 對應的 Zod schema 在 `${FE_SCHEMAS_DIR}/` 下 export。
- 所有 contract 對應的 API client 函式（pure fetch + Zod parse return）存在於 `src/lib/api/`。
- 所有 fixture mock handler 對應的 `page.route(url, handler)` 已註冊；handler 內**入向 + 出向**皆透過 Zod schema parse（boundary invariant I2）。
- 所有 `L4.source_refs.component` 指向的 Story export 存在且帶 accessible-name args；對應的元件 skeleton 可 render、互動目標（按鈕 / input / link）可被 step def 定位並接收 user event；onClick / onSubmit / onChange 等 handler **prop 簽名**存在（內部是否呼叫 api-client 由 Green 補）。
- API host 與 Next.js dev host 不衝突。

通過 Pre-Red 後，下列**行為性事實 _不_ 必須成立**：

- 元件做的事是對的（state 變化 / render 內容 / 條件分支）→ Green 補。
- API client 業務分支處理（retry / error UI mapping / cache）→ Green 補。
- Mock handler 對非 happy-path 回應的決策 → Green 在 scenario 內透過 `mockApi.override` 設置。

換言之：Pre-Red 把所有「**讓 step body 能跑到正確失敗點**」的 wiring 接通；Red 跑出來的紅必定紅在「業務邏輯沒做」這一面，而非「scaffolding 還沒長」。

## §0.2 三種典型「越界 GREEN」反例（NORMATIVE）

§0 的「行為性事實不必須成立」常被 worker 在落 wiring 時誤踩。下列三種寫法在 Pre-Red gate 判 GO 之前**禁止**出現。它們把 GREEN Wave 1 該補的「業務行為」搶先實作，會讓第一輪 RED **假綠**（紅應紅在「行為缺失」這一面，卻因 wiring 自動把資料流接通而綠掉），破壞合法紅燈定義（§1）。

判別總則：worker 若在寫 wiring 時是「順著 feature 的 expected outcome 把資料流接通」，幾乎必越界。Pre-Red 寫的每一行都要回答：「拿掉這行，step body 還能跑到失敗點嗎？」能 → 越界；不能 → 邊界。

### §0.2.1 Mock handler — 禁止對 closure store 做業務 mutation

**越界（Pre-Red 不可寫）**：

```ts
await page.route(`${API_HOST}/games/*/actions`, async (route) => {
  const game = store.get(gameId)!;
  const stub = nextResponse(opId);
  // ⛔ 越界：把 stub 套用回 closure store → 後續 mock-state-then 自動變綠
  applyMutations(game, stub); // 改寫 game.players[].hp / eliminated / boss.password
  const merged = { ...stub, gameId };
  await route.fulfill({ /* ... */ body: JSON.stringify(merged) });
});
```

**邊界（Pre-Red 合法）**：

```ts
await page.route(`${API_HOST}/games/*/actions`, async (route) => {
  const stub = nextResponse(opId);             // 取 stub
  if (!stub) { /* 500 */ }
  // ✅ 不呼叫 applyMutations()；closure store 不變
  const merged = { ...stub, gameId, actionType };
  const body = respSchema.parse(merged);       // I2 出向 Zod gate（強制）
  await route.fulfill({ status: 200, contentType: 'application/json', body: JSON.stringify(body) });
});
```

**判準**：mock handler 在 Pre-Red 階段只負責「取 stub → 通過 Zod parse → fulfill」。下列動作都屬 GREEN Wave 1：
- 依 stub 內容反寫 closure store（`applyMutations` / `mutate*` / `apply*` helper）
- 依 request body 推導下一輪 mock state（per-call sequence 算 HP）
- 對非 happy-path 的條件分支（屬 §0 行為性事實第三條，由 step def 在 scenario 內透過 `mockApi.override(...)` 設置）

靜態檢查線索：`page.route` handler 內若出現 `store.<set|delete|clear>(` 以外的 closure-state 寫入呼叫，列為候選越界，寫入 Pre-Red 報告。

### §0.2.2 Event handler — 禁止把 API response 流回 component state

**越界（Pre-Red 不可寫）**：

```tsx
const onGroupAttack = async () => {
  const res = await executeBossGroupAttack(gameId, { guesses });
  // ⛔ 越界：response → local state 的 mapping 是 GREEN Wave 1 該補的業務 wire-up
  setSnapshot((prev) => mergeAttack(prev, res));
};
```

**邊界（Pre-Red 合法）**：

```tsx
const onGroupAttack = async () => {
  await executeBossGroupAttack(gameId, { guesses });
  // ✅ 只 fire API call；不把 response 寫回 state。snapshot 保持 initial。
};
```

**判準**：Pre-Red 階段 event handler 的 prop 簽名（onClick / onSubmit / onChange）**必須**存在，handler body **可**呼叫 api-client function（讓 `api-call-then` 等 Tier-2 step 在 GREEN 觀察得到 call site），但**不可**對 component local state 做 response-driven mutation：
- 合法：`await client.xxx(...)`、`await client.xxx(...).catch(noop)`
- 越界：`setX(res)` / `setX((prev) => merge(prev, res))` / `dispatch({ type, payload: res })` / 任何 `useState` setter 收 response 衍生值

合法例外（不算越界）：載入時的初始 snapshot 取得，例如
```tsx
useEffect(() => { getGame(id).then(setSnapshot); }, [id]);
```
這屬於 §0 結構性事實「skeleton 能 render 出來」的最小需求；即使拿掉，按鈕仍在 accessibility tree 內可被 locator 命中，故保留無害。

### §0.2.3 Conditional render — 禁止以業務狀態 gate 互動目標 / 終態 UI

**越界（Pre-Red 不可寫）**：

```tsx
return (
  <main>
    {/* ⛔ 越界：用 gameStatus 決定 modal 出現與否 = GREEN 才該補的條件分支 */}
    <GameOverModal open={snapshot.gameStatus !== "in_progress"} />
  </main>
);
```

**邊界（Pre-Red 合法）**：

```tsx
return (
  <main>
    {/* ✅ open 寫死 false；或整段先不 render，GREEN 再加回 */}
    {/* 第一輪 RED 必紅在「assert dialog visible but not found」— 合法業務紅 */}
    <GameOverModal open={false} />
  </main>
);
```

**判準**：所有「依 state 決定 mount / unmount / visibility / disabled / enabled」的條件式都屬 GREEN Wave 1。Pre-Red 兩個合法做法擇一：
- 把元件 render 寫死關閉態（modal `open={false}`、badge `hidden` 固定 true、button `disabled` 固定 false）；
- 不 render（適用於不影響 locator 命中的 end-state element，例如 `<GameOverModal>` 整段拿掉，GREEN 補時再加回）。

**禁止**：把任何 `snapshot.*` / `state.*` / `props.derived` 放進 JSX 條件位（`{cond && <X />}` / `<X visible={cond}>` / `disabled={derivedFromState}`）。

合法例外（不算越界）：
- list 本身的 render（如 `players.map(p => <PlayerHpBar slot={p.slot} hp={p.hp} />)`）— list 是「skeleton 結構」，內部直傳初始值 `hp={p.hp}` 因 snapshot 維持 initial 故定格在初始值（例：50/50/50），第一輪 RED 紅在「assert 30 but got 50」這條合法業務紅。
- 純展示初始值（`<PasswordDisplay password={boss.password} />` 在 snapshot 不變時等同寫死）。
- loading skeleton 切換，但條件變數**不**依賴 API response（如 `useState(false)` 全程不改）。

---

預期落地後第一輪 RED 訊號（典型 5-scenario 範例）：

| Scenario | 預期紅燈 | 越界若未拆會發生什麼 |
|---|---|---|
| HP 條同步更新 | ❌ assert "30" but got "50" | 若 mergeAttack 沒拆 → 綠 |
| 1A2B 結算 | ❌ assert "35" but got "50" | 同上 |
| HP 歸零出局 badge | ❌ badge not visible | 若 applyMutations 沒拆 → 綠 |
| 全員出局 modal | ❌ dialog not found | 若 `open={cond}` 沒寫死 false → 綠 |
| 魔王 HP=0 modal | ❌ dialog not found | 同上 |

若上述任一條意外綠燈，**先回頭找哪一段越界 wiring 還沒拆**，再進 RGB cycle。

## §1 為什麼存在

**合法紅燈** = production 行為缺失，導致 acceptance 斷言或預期例外失敗。

對 nextjs-playwright 而言，下列七種「**基礎設施紅**」會在 step body 真正執行前把 Red 打掛：

1. **OpenAPI ↔ Zod schema drift** — api.yml 加了新 schema，`${FE_SCHEMAS_DIR}/` 沒同步 → fixture / api-client 一進 parse 就被 Zod 擋下。
2. **Schema ↔ api-client drift** — api.yml 有 op，api-client 沒對應函式 → 元件無從呼叫。
3. **api-client ↔ fixture drift** — api-client 打的 URL 在 fixture 沒 `page.route` 攔截 → 真打網路 / 命中 dev server 404。
4. **Fixture 缺 Zod parse gate（I2 違反）** — `route.fulfill` 直接 `JSON.stringify(arbitraryShape)`，未經 schema parse → contract drift 偷渡進綠燈。
5. **Storybook contract gap** — Story export 不存在 / 缺 a11y args；或元件 skeleton 沒 render 出互動目標（按鈕/input 不在 DOM 或不在 accessibility tree）→ step def click 不到。
6. **API host 與 Next.js dev host 重疊** — `page.route` glob 攔到 page navigation。
7. **Tier-2 handler 未 enable** — `dsl.yml` 用了 Tier-2 但 `test-strategy.yml` 沒宣告。

→ 本 Hook 為 **Pre-Red gate**：依依賴順序收斂 7 條，再進 RGB cycle。

## §2 觸發條件

- profile skill 為 `/aibdd-form-api-spec` 但 `${CONTRACTS_DIR}/api.yml` 不存在 → STOP，route `/aibdd-form-api-spec`（屬 Plan phase 任務）。
- 存在 `${CONTRACTS_DIR}/api.yml` → §3.1–§3.4 必須執行。
- `api.yml`、`${FE_SCHEMAS_DIR}/*.ts`、`src/lib/api/*.ts`、`${FE_FIXTURES_FILE}` 任一自上次 sync 後變更 → 視為過期，**重新**執行 §3。
- 目標 Scenario 有 `L4.source_refs.component` 指向新 Story export → §3.5 強制重跑。

## §3 流程（依依賴順序）

依賴鏈：`api.yml → schemas → api-client → fixture → component+story`。任一環缺，下游全 NO-GO。

### §3.1 OpenAPI truth（Plan-phase 入口檢查）

來源：`${CONTRACTS_DIR}/api.yml`（Plan phase 產出，本 phase **不撰寫 / 不修改**）。

解析並對齊：

- `paths.*` × `(method)` → `operationId`
- `components.schemas.*`（request / response）
- 必填 / 選填 / nullable 欄位
- enum / oneOf / discriminator

ASSERT 每條 `path × method` 都帶 `operationId`；缺者 NO-GO，route Plan phase 補回 `/aibdd-form-api-spec`。

### §3.2 Zod schemas — 與 api.yml 對齊

掃描：`${FE_SCHEMAS_DIR}/**/*.ts`

規則：

- 每個 OpenAPI `components.schemas.<X>` 對應一個 `export const <X> = z.object({...})`（命名與 OpenAPI 一致）。
- 欄位 type、`.optional()` / `.nullable()`、enum 與 OpenAPI 一致（`required` 陣列 + `nullable: true` 雙軸）。
- `z.infer<typeof <X>>` 為下游 props / fixture / mock 唯一 typed entry；**禁止**手寫對應 `interface`。
- **路徑 A（推薦）— codegen**：使用 `openapi-zod-client --export-schemas` 產生 `${FE_SCHEMAS_DIR}/_generated.ts` 作為 SSOT；本檔禁止手改，由 codegen drift CI gate（`pnpm schemas:gen && git diff --exit-code`）守護。
- **路徑 B（user-elected）— 手寫**：若專案明確選擇先不導入 codegen，由作者依本節規則手抄 Zod；drift → §4.1 路徑 B。兩條路徑同時遵守上述 schema 對齊規則，差別僅在誰寫。

### §3.3 API client interfaces — 函式骨架對齊（NEW）

掃描：`src/lib/api/**/*.ts`（建議 per-aggregate 切檔，如 `src/lib/api/room.ts`）。

規則：

- 每個 api.yml `operationId` 對應一個 export 函式；input / output 型別由 §3.2 schema 推導（`z.infer`）。
- 函式體 = 一行 `apiFetch(path, { method, body, schema })`，由 `src/lib/api-client.ts` wrapper 統一處理 base URL、headers、`!res.ok → ApiError`、Zod `safeParse`。**禁止**：retry、try/catch 業務分支、cache、conditional fallback、auth 自動續期、繞過 wrapper 直寫 `fetch()` — 那些都是 Green 內容。
- base URL 由 wrapper 透過 `src/lib/env.ts` 統一讀 `NEXT_PUBLIC_API_BASE_URL`（client）/ `API_BASE_URL`（server），**不得**在 api function 內寫死 host 或重抓 `process.env`。
- 缺對應函式 → NO-GO，§4.2。

合法骨架範本：

```ts
import { apiFetch } from '@/lib/api-client';
import { CreateRoomRequest, CreateRoomResponse } from '@/lib/schemas/_generated';
import type { z } from 'zod';

export async function createRoom(
  input: z.infer<typeof CreateRoomRequest>,
): Promise<z.infer<typeof CreateRoomResponse>> {
  return apiFetch('/rooms', {
    method: 'POST',
    body: input,
    schema: CreateRoomResponse,
  });
}
```

### §3.4 Fixture mock handlers — Zod parse 雙向 gate（I2 落地）

掃描：`${FE_FIXTURES_FILE}`（預設 `features/steps/fixtures.ts`）

規則（boundary invariant I1 + I2 落地）：

- 每個 plan DSL entry 中 surface == `api` 的 operation 對應一個 `page.route(url_pattern, handler)` 匹配（method + path）。
- Handler 內部**必須**：
  - 入向（POST/PUT/PATCH 等帶 body op）：`<Op>Request.parse(req.postDataJSON())`
  - 出向：`<Op>Response.parse(<body>)` 在 `route.fulfill(...)` 之前
- 任一 parse 缺漏 → **NO-GO（I2 違反）**，§4.3。靜態檢查：`check_mock_schema_gate.py` 對每個 `route.fulfill(...)` 上方同 block 範圍找對應 `<Op>*.parse(`。
- mock 狀態必須住在 fixture closure（`Map<>` / object / class）。**禁止** import `src/mocks/*` 或任何產品端 module 作為 store；**禁止**透過 HTTP `/__test__/*` 端點跨 process 同步。
- fixture 必須對外暴露純 in-process 控制 API（`mockApi.seedXxx(...)` / `mockApi.override(...)` / `mockApi.reset()`）供 step defs 同步呼叫；**禁止**包成 async HTTP 呼叫。
- handler 預設回 happy-path response；錯誤路徑由 step def 透過 `mockApi.override(operationId, response)` 在 scenario 內 per-call 設置，**不**在 default handler 寫死。
- **handler body 內禁止對 closure store 做業務 mutation**（`applyMutations` / `mutate*` / `apply*` helper）— 見 §0.2.1。Pre-Red mock 只做「取 stub → Zod parse → fulfill」。
- 缺 page.route handler 但目標 Scenario 觸發該 op → NO-GO，§4.3。
- 多餘 handler（DSL 已不存在的 op）→ 記為 drift／報告，**不**單獨阻擋 GO。

### §3.5 Component skeleton + Story（colocated）

來源：dsl.yml 中所有 `L4.source_refs.component` 條目。

規則（boundary invariant I4 落地）：

- anchor 形式 = `<file>.stories.@(ts|tsx)::<ExportName>`；缺 `::Export` 後綴 → NO-GO。
- Story export 必須在 file 內存在（檔案掃描）。
- Story export 的 `args` 必須宣告 accessible-name（`children` / `label` / `aria-label` / `name` 任一）；缺 → NO-GO，**不自動修正**（route 給 Story 作者，§4.4）。
- **Story file 路徑規則**：colocate 於 `src/components/<X>/<X>.stories.@(ts|tsx)`（隨元件同層）；**淘汰**集中式 `src/stories/`。Storybook glob 對應 `'../src/components/**/*.stories.@(ts|tsx)'`。
- **Component skeleton 結構性要求**（讓 Red 能定位互動目標、執行 click，把紅燈推到合法業務位置）：
  - 元件 export 存在、可 render、不 throw 在 mount。
  - Story 範例下，UI 互動目標（按鈕 / input / link）能被 step def 透過 role / label / test-id 找到並接收 user event（click / fill）。
  - onClick / onSubmit / onChange 等 handler **prop 簽名**存在（即使內容是 `() => {}` 也算合法 — handler 內**是否呼叫 api-client**屬於業務 wire-up，是 Green 的內容）。
  - **反例（pre-red 該擋）**：元件根本沒 render 出按鈕、按鈕被無條件 disabled、element 不在 accessibility tree → step 連 click 都點不到 → false red（紅在 scaffolding，非業務）。
  - **本 phase 不做的事**：強制 onClick 內部接 api-client；那是 Red 第一輪紅出來後 Green 該補的最小 wire-up，Red 紅在「click 後沒 API call fire」是合法紅。
  - **本 phase 也不做的事**：把 API response 流回 component state（`setX(res)` / `setX((prev) => merge(prev, res))`）— 見 §0.2.2；把 `snapshot.*` / 衍生 state 放進條件 render（`<Modal open={cond}>` / `{cond && <X />}`）— 見 §0.2.3。end-state UI（modal / badge / 出局標記）一律寫死關閉態或不 render。
- 若元件來自 design-system library → **必須**先呼 `${PROJECT_SLUG}-sb-mcp`（`list-all-documentation` → `get-documentation`）確認 prop 支援；MCP 查不到 → NO-GO。

### §3.6 Tier-2 enablement match

來源：`dsl.yml` 中所有 `L4.preset.handler` × `${TEST_STRATEGY_FILE}.tier2_handlers`。

規則：

- 每個 Tier-2 handler 在 dsl.yml 出現 → `test-strategy.yml.tier2_handlers.<handler>` 必須為 `enabled`。
- 不一致 → NO-GO，§4.5。

### §3.7 API host separation

規則（boundary invariant I1 落地之物理層保證）：

- API base URL 之 host 必須與 Next.js dev server host **不同**，避免 `page.route` glob 同時 intercept page navigation。
- 預設配置：Next.js dev = `http://localhost:3000`（`playwright.config.ts.use.baseURL`）；API = `http://localhost:4000`（`NEXT_PUBLIC_API_BASE_URL`）。
- `.env.local` 必須設 `NEXT_PUBLIC_API_BASE_URL` 為非 Next.js dev host。
- `playwright.config.ts.webServer.env` **不得**覆寫 `NEXT_PUBLIC_API_BASE_URL` 為 Next.js dev host；建議完全不設 `webServer.env`，讓 `.env.local` 預設生效。
- `${FE_FIXTURES_FILE}` 中 `page.route(...)` 之 URL pattern 必須以該 API host 為前綴，確保 page navigation 不被攔截。
- 任一條違反 → NO-GO，§4.7。

### §3.8 GO / NO-GO

| 狀態 | 行動 |
|---|---|
| §3.1–§3.7（適用範圍）全綠 | **GO** — 進 Red |
| 缺 Zod schema / api-client / page.route handler，且能由 OpenAPI 安全 autogenerate | **自動修正**後 **GO**（§4.1 / §4.2 / §4.3） |
| Story 缺 a11y args / 元件 skeleton 缺 onClick prop 簽名或互動目標不在 accessibility tree / Tier-2 未 enable / API host 衝突 / OpenAPI 自身有歧義 | **NO-GO** — 暫停並產出報告，交由人工 |

## §4 修正流程（nextjs-playwright）

### §4.0 環境前置

凡將執行 `npm install` / `npm run dev` / Playwright `webServer` / `npm run storybook` / `npm run build`：

1. **查文件**：先對照 `package.json` scripts 與 `playwright.config.ts` 的 `webServer` 段；**禁止**只靠記憶猜 port 或 cmd。
2. **預設假設**：walking skeleton 在 frontend 根附 `package.json`、`playwright.config.ts`、`.env.example`，dev server 預設 `http://localhost:3000`。
3. **強制步驟**：
   - **①** `npm install`（或本專案 `dev-constitution` 明載之等價指令）。
   - **②** 確認 port 3000 未被占用；占用 → 改 port 並同步 `playwright.config.ts.use.baseURL`。
   - **③** 確認 `.env.local` 之 `NEXT_PUBLIC_API_BASE_URL` 非 Next.js dev host（§3.7）；若被 `playwright.config.ts.webServer.env` 覆寫 → 移除。
   - **④** 若 `npm install` 失敗、port 釋放不來、`.env` 缺鍵、或 dev server 起不來 → **立刻 STOP（NO-GO）**，向使用者**白話說明**缺什麼、要裝什麼，待修復後再續；**不**繼續 §4.1–§4.7。

### §4.1 補 Zod schema

1. 讀取對應 OpenAPI `components.schemas.<name>` 結構。
2. **路徑 A — codegen**：跑 `pnpm schemas:gen`（`openapi-zod-client --export-schemas` 落到 `${FE_SCHEMAS_DIR}/_generated.ts`）。
3. **路徑 B — 手寫**：在 `${FE_SCHEMAS_DIR}/<aggregate>.ts` 寫入 `export const <Name> = z.object({...})`；nullable / optional 邊界要對齊 OpenAPI 雙軸（`required` 陣列 + `nullable: true`）。
4. 跑 `tsc --noEmit` 與 §3.2 確認綠燈。

### §4.2 補 API client function（NEW）

1. 從 api.yml 抓 op 的 path / method / requestBody / responses。
2. 在 `src/lib/api/<aggregate>.ts` 加一個 export 函式，套 §3.3 範本。
3. **禁止**寫 retry / cache / 業務分支；只 call `apiFetch(path, { method, body, schema })`。wrapper（`src/lib/api-client.ts`）已負責 base URL、headers、Zod parse、`!res.ok → ApiError` 拋擲；api function 端不得重複實作。
4. 跑 `tsc --noEmit`，重跑 §3.3。

### §4.3 補 page.route handler + Zod parse gate

1. 在 `${FE_FIXTURES_FILE}` 的 `mockApi` fixture closure 內部 store（`Map<>` / object / class）加上對應 entity。
2. 加 `page.route(url_pattern, handler)`：
   - dispatch by `req.method()` + `url.pathname`
   - **入向**：若有 body，先 `<Op>Request.parse(req.postDataJSON())`
   - mutate closure-local store
   - build response object
   - **出向**：`<Op>Response.parse(<body>)`（必行；禁止省略）
   - `route.fulfill({ status, contentType: 'application/json', body: JSON.stringify(<parsed>) })`
3. **預設回 happy-path**；錯誤路徑由 step def 透過 `mockApi.override(...)` 在 scenario 內 per-call 設置，**不**在 default handler 寫死。
4. 跑 `tsc --noEmit` + `check_mock_schema_gate.py`，重跑 §3.4。

### §4.4 Storybook binding 修正（manual gate）

Story 缺 `accessible-name args` → **禁止**自動補（自動補出來的 a11y 名稱常是錯的，會在 Red 通過但 a11y 實際失格）。

1. 列出每個缺 args 的 Story export，附「目前 args」與「建議補上的 args」候選。
2. EMIT 報告，route 給 Story 作者；NO-GO 暫停。
3. 作者補完 → 重跑 §3.5。

元件 skeleton 缺 onClick / onSubmit / onChange handler prop 簽名，或互動目標不在 accessibility tree → manual gate；列出缺漏的 prop / element，由作者補回 prop 簽名（**禁止**自動補進 api-client 呼叫 — 那是 Green 內容）。

### §4.5 Tier-2 enablement（manual gate）

1. 列出 `dsl.yml` 用了但 `test-strategy.yml` 未 enable 的 Tier-2 handler。
2. EMIT 報告，問 user：要 enable 還是改 handler？
3. user 決定 → 對應更新 `test-strategy.yml.tier2_handlers.<handler>` 或 `dsl.yml`，重跑 §3.6。

### §4.6 OpenAPI rename / breaking change（manual gate）

OpenAPI 把 `operationId` 從 X 改成 Y → autogenerate 會看成「drop X + add Y」，下游 schema / api-client / mock / step file 全部 rename 風險高。**禁止**自動 rename；需人工 revision，並由 `/aibdd-reconcile` 決定是否 cascade plan。

### §4.7 API host fix

1. 確認 `.env.local` 設 `NEXT_PUBLIC_API_BASE_URL` 為非 Next.js dev host（如 `http://localhost:4000`）。
2. 移除 `playwright.config.ts.webServer.env` 內任何 `NEXT_PUBLIC_API_BASE_URL` 覆寫。
3. 確認 `${FE_FIXTURES_FILE}` 的 `page.route(...)` URL pattern 以該 API host 為前綴。
4. 重跑 §3.7。

## §5 失敗處理

- §4.0 環境未通（`npm install` / port / `.env` / dev server）→ **STOP**，依 §4.0④ 向使用者說明；**不**視為 RGB 失敗點數，亦不強行 autogenerate。
- `tsc --noEmit` 失敗 → 還原最近一筆 schema / api-client / fixture 修改，**NO-GO** 並報告。
- `check_mock_schema_gate.py` 報告 I2 違反 → 視為 §3.4 NO-GO，§4.3 補上缺漏 `.parse(...)`。
- `api.yml` 無法解析 → **NO-GO**，回到 Plan phase `/aibdd-form-api-spec`。
- §3.5 Storybook precheck 連續 NO-GO 兩輪 → **STOP** 並 route `/aibdd-reconcile`（檢查 plan 是否誤指 Story export）。
- `${PROJECT_SLUG}-sb-mcp` MCP 工具不可用（未安裝 / token 失效）→ **STOP**，向使用者說明缺什麼、要裝什麼；**不**繞過 MCP 用命名直覺臆測 props。

## §6 與 RGB cycle 的關係

contract / schema / api-client / mock-fixture / component-story / api-host precheck **不計入** RGB step 編號；為 Pre-Red gate。

- **通過**後才開始計 `R(1) → G(1) → B(1)`。
- **未通過**則不啟動 Red 主流程；Red loop budget **不消耗**於「合法紅燈」計數。
- Pre-Red gate 自身有 budget（§4.1 / §4.2 / §4.3 自動修正 ≤ 3 輪）；超過視為環境不穩，STOP 並 route `/aibdd-reconcile`。
