## §3.3 page.route fixture handler（每個 plan DSL operation 一個）

由 GREEN Wave 1 在 `features/steps/fixtures.ts` 寫入：

```ts
const API_HOST = 'http://localhost:4000'; // distinct from Next.js dev (localhost:3000)

export const test = base.extend<{ mockApi: MockApi }>({
  mockApi: async ({ page }, use) => {
    const store = new Map<string, RoomState>();

    await page.route(`${API_HOST}/<resource>/**`, async (route) => {
      // dispatch by req.method() + URL path
      // mutate `store`; route.fulfill({ status, contentType, body })
      // every fulfillment must conform to its responseSchema (Zod)
    });

    const api: MockApi = {
      reset: () => store.clear(),
      seed<Entity>: (input) => store.set(/* ... */),
      inspect<Store>: (where) => /* read from store */,
    };
    await use(api);
  },
});
```

**規則**：
- mock 狀態必須住 fixture closure，**禁止** import `src/mocks/*`（不應存在）或產品端 module。
- 每個 plan DSL surface == `api` 的 operation 必須對應一個 `page.route` 匹配（method + path）。
- handler 回傳 JSON 必須能被對應 `responseSchema.parse(...)` 通過（boundary invariant I2）。
- **handler body 不得對 closure store 做業務 mutation**（無 `applyMutations` / `mutate*` / `apply*` 等 helper 呼叫）— 見 `prehandling-before-red-phase.md` §0.2.1。
- per-scenario reset 由 fixture scope（test）自動處理；step defs **不**手動 reset。

**Wave 1 越界檢查（worker self-audit）**：寫完 fixture / page 後逐項確認：

| 位置 | 越界訊號 | 邊界寫法 | 規範來源 |
|---|---|---|---|
| `fixtures.ts` page.route handler | 呼叫 `applyMutations(store, stub)` 等 helper 改寫 closure | 只 `respSchema.parse` 後 `route.fulfill` | §0.2.1 |
| `page.tsx` event handler | `setX(res)` / `setX((prev) => merge(prev, res))` | `await client.xxx(...)` 就結束 | §0.2.2 |
| `page.tsx` JSX | `<Modal open={snapshot.gameStatus !== ...}>` / `{cond && <X />}` | `<Modal open={false}>` 或整段不 render | §0.2.3 |
| `page.tsx` initial load | n/a（合法） | `useEffect(() => getX(id).then(setX), [id])` | §0.2.2 例外 |

## §3.5 Storybook binding（Story export with accessible-name args）

每個 plan DSL 內 `L4.source_refs.component` 指到的 Story export 必須存在於對應 `*.stories.@(ts|tsx)` 檔。Story 的 `args` 必須宣告 accessible-name（`children` / `label` / `aria-label` / `name` 任一）。

## §3.6 Tier-2 enablement

每個 dsl.yml 用到的 Tier-2 handler（`api-stub` / `url-then` / `api-call-then` / `mock-state-then`）必須在 `specs/frontend/test-strategy.yml` 標記為 `enabled`：

```yaml
tier2_handlers:
  api-stub: enabled        # 用到 boundary mock per-scenario behavior override 時
  url-then: enabled        # 用到 URL pathname / query / hash assertion 時
  api-call-then: disabled  # 尚未用到 → disabled
  mock-state-then: disabled
```

**未來改進**：`/aibdd-plan` Phase 6 渲染 `test-strategy.yml` 時應自動寫入此 section（依 dsl.yml 內 Tier-2 handler 用量推導）。

## §3.7 API host separation

API base URL 之 host 必須與 Next.js dev server host **不同**，避免 `page.route` glob 同時 intercept page navigation。

**檢查項**：
- `.env.local` 設 `NEXT_PUBLIC_API_BASE_URL=http://localhost:4000`（或任何非 `localhost:3000` 之 host）。
- `playwright.config.ts.webServer.env` **不得**覆寫 `NEXT_PUBLIC_API_BASE_URL`（建議不設 `webServer.env`）。
- `features/steps/fixtures.ts` 的 `page.route(...)` URL pattern 以該 API host 為前綴。

starter template 已預配（v2）：
- `.env.example` 預設 `NEXT_PUBLIC_API_BASE_URL=http://localhost:4000`
- `playwright.config.ts` 不含 `webServer.env`
- `playwright.config.ts.use.baseURL` 為 `http://localhost:3000`（Next.js dev）

新專案理論上 `cp .env.example .env.local` 即生效；如手動改過需確認。
