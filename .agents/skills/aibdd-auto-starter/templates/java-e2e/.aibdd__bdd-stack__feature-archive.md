# Feature Archive Runtime

## Arguments anchor (not a free-form label)

Archive layout uses the **same** function-package segment as spec truth. Everything below is defined **only** by keys declared in the project arguments file at **`${AIBDD_ARGUMENTS_PATH}`** — **no** external template path, **no** hard-coded concrete config path in this instruction file (use the bound path token only).

**Path shape (self-contained contract):** `TRUTH_FUNCTION_PACKAGE` 在 `arguments.yml` 中以借位形態 `${TRUTH_BOUNDARY_PACKAGES_DIR}/<<NN-functional-module>>` 寫入；caller-context 在 invoke skill 時提供具體 functional module slug 後，`TRUTH_FUNCTION_PACKAGE` 解析為一個 **function package root** 的具體 directory — 通常是 `.../packages/<NN-slug>/` 路徑下、`${TRUTH_BOUNDARY_PACKAGES_DIR}` 的直接子目錄。`FEATURE_SPECS_DIR` **MUST** expand as `${TRUTH_FUNCTION_PACKAGE}/features`. These expansions are the only definitions this file needs.

| Key | Role |
|-----|------|
| `AIBDD_ARGUMENTS_PATH` | Resolve-time path to the YAML file that **declares** the keys in this section; **never** hard-code a filesystem path in prose — bind **`${AIBDD_ARGUMENTS_PATH}`** only. |
| `TRUTH_FUNCTION_PACKAGE` | 借位形態（slot literal `<<NN-functional-module>>`）；caller-context 提供 functional module slug 後解析為具體 directory path。 |
| `FEATURE_SPECS_DIR` | `${TRUTH_FUNCTION_PACKAGE}/features` — Gherkin truth for that package. |
| `TRUTH_BOUNDARY_PACKAGES_DIR` | Boundary `packages` directory; each **child directory name** is one function package folder on disk. |
| `JAVA_TEST_FEATURES_DIR` | Cucumber feature root under the product tree（本 starter 鎖定 `src/test/resources/features`，沒有獨立子鍵改寫此路徑）。 |

**`<current_spec_package>`** in this file means **`basename(${TRUTH_FUNCTION_PACKAGE})` after resolution** — the final path segment of `TRUTH_FUNCTION_PACKAGE`, i.e. the same name as `${TRUTH_BOUNDARY_PACKAGES_DIR}/<current_spec_package>/` on disk. It is **not** a separate YAML key; use **`TRUTH_FUNCTION_PACKAGE`** as the bound parameter and derive the segment.

## archive mechanism

Java E2E starter expects executable `.feature` files under `src/test/resources/features/`, **organized by mandatory `basename(${TRUTH_FUNCTION_PACKAGE})` / `<current_spec_package>/` matching the specs function package directory**. It does not automatically copy rule-only specs from `${FEATURE_SPECS_DIR}` into the runtime tree.

### RED phase: archive so Cucumber sees the failing scenario

During **RED** (`/aibdd-red`), the runner **MUST** load the executable `.feature` from `src/test/resources/features/**/*.feature` (same glob contract as **Acceptance Runner Runtime**: `RunCucumberTest` → `@SelectClasspathResource("features")`). Spec truth stays under `${FEATURE_SPECS_DIR}` (inside `${TRUTH_FUNCTION_PACKAGE}`). **Do not** point Cucumber straight at the entire specs tree mixed with `@ignore` rule-only files unless your project explicitly narrowed `@SelectClasspathResource`; the starter contract is **mirror into `src/test/resources/features/<current_spec_package>/`**.

**Preferred mechanism:** **symbolic link** (file or directory) from the runner subtree → the spec-side file or `${FEATURE_SPECS_DIR}` so **one physical file** remains SSOT and RED edits land where Discovery / plan expect them.

**Prepare layout**

1. Resolve `FEATURE_SPECS_DIR` 與 `basename(${TRUTH_FUNCTION_PACKAGE})` → `<current_spec_package>` from `${AIBDD_ARGUMENTS_PATH}`。
2. Create the package mirror directory once: `src/test/resources/features/<current_spec_package>/`.
3. For each executable feature that must run in RED, add a **symlink** (below) **or**, if symlinks are impossible on that machine, **copy** the file and accept manual re-sync (last resort)。

**Per-file symlink (typical)**

Replace `<NN>-<title>.feature` with the real filename; substitute concrete paths after resolving the `${...}` keys (same repo clone, absolute or relative — **relative symlinks survive** moves of the repo root better on laptops).

**POSIX (macOS / Linux / Git Bash)**

```bash
mkdir -p "src/test/resources/features/<current_spec_package>"
ln -sfn "${FEATURE_SPECS_DIR}/<NN>-<title>.feature" \
  "src/test/resources/features/<current_spec_package>/<NN>-<title>.feature"
```

**Windows — PowerShell** (symlink creation needs **`SeCreateSymbolicLinkPrivilege`** OR **Developer Mode** on Windows 10+ so non-admin shells can create symlinks; otherwise run elevated once or use the fallback block below)

```powershell
New-Item -ItemType SymbolicLink `
  -Path "src/test/resources/features/<current_spec_package>/<NN>-<title>.feature" `
  -Target "${FEATURE_SPECS_DIR}/<NN>-<title>.feature" `
  -Force
```

**Windows — Command Prompt (`cmd`)** (same privilege / Developer Mode caveat)

```bat
mklink "src\test\resources\features\<current_spec_package>\<NN>-<title>.feature" "${FEATURE_SPECS_DIR}\<NN>-<title>.feature"
```

**Directory symlink (when the whole `${FEATURE_SPECS_DIR}` tree for that package should appear under the runner)**

- POSIX: after `mkdir -p "src/test/resources/features"`, remove empty placeholder dir if needed, then
  `ln -sfn "${FEATURE_SPECS_DIR}" "src/test/resources/features/<current_spec_package>"`
  **only if** that directory name is **exclusively** Cucumber-visible executable features (no rule-only pollution); otherwise prefer **per-file** links.
- Windows: `mklink /D` for a **directory** symlink, or PowerShell `New-Item -ItemType SymbolicLink` with a directory target — same privilege rules as file symlinks.

**Fallback when symlinks are blocked** (locked-down Windows without Developer Mode, some CI sandboxes, or FAT/exotic volumes)

1. **Copy**: POSIX `cp -f "${FEATURE_SPECS_DIR}/<NN>-<title>.feature" "src/test/resources/features/<current_spec_package>/"` — re-run copy whenever the spec file changes (drift risk).
2. **PowerShell copy**: `Copy-Item -Force ...`
3. **Forbidden as archive substitute for this starter:** inventing a second edited `.feature` only under `src/test/resources/features` without tying it back to `${FEATURE_SPECS_DIR}` — that **always** breaks SSOT。

**RED visibility check (after link or copy)**

```bash
./mvnw test -Dcucumber.execution.dry-run=true
```

Confirm the scenario meant for RED appears in the collected steps list; if not, fix the link target or `<current_spec_package>/` path **before** claiming RED.

## spec feature source

`${FEATURE_SPECS_DIR}` is the accepted behavior truth location once caller-context provides the functional module slug that resolves the `<<NN-functional-module>>` borrow. The folder name under `${TRUTH_BOUNDARY_PACKAGES_DIR}/<current_spec_package>/` (the segment after `.../packages/`) is the **canonical** name for runtime layout — **identical** to `basename(${TRUTH_FUNCTION_PACKAGE})` for the bound package。

## runtime feature target

Executable product features MUST NOT sit loose at `src/test/resources/features/` 根目錄。They MUST be archived under：

```text
src/test/resources/features/<current_spec_package>/<NN>-<title>.feature
```

`<current_spec_package>` is mandatory and **MUST** be the **same directory name** as the specs function package folder: `.../packages/<current_spec_package>/`（same basename as `${TRUTH_FUNCTION_PACKAGE}` when that path is resolved）。

Parallel step definitions MUST live under：

```text
src/test/java/${BASE_PACKAGE_PATH}/steps/<current_spec_package>/
```

using that **identical** `<current_spec_package>` — **never** derive the runtime folder name only from step-def layout; specs package folder name is SSOT。

Walking-skeleton `src/test/resources/features/HealthCheck.feature` MAY stay **at features root only** — it is the single allowed root-level feature and is **not** product BDD。

## include policy

- Rule-only Discovery features remain under specs and are marked `@ignore`。
- Executable acceptance features must be generated or archived into `src/test/resources/features/<current_spec_package>/` only after Examples / executable scenarios exist；**never** park new product `.feature` files next to `HealthCheck.feature` at root。
- Starter smoke `HealthCheck.feature` is a walking skeleton exception and is not product BDD。

## visibility check

Run `./mvnw test -Dcucumber.execution.dry-run=true` and confirm the target feature is discovered。

## forbidden

- Do not let Cucumber scan rule-only specs directly。
- Do not archive a feature without executable scenarios。
- Do not place new product `.feature` files directly under `src/test/resources/features/` 根目錄（violates `<current_spec_package>` rule）；root is reserved for skeleton `HealthCheck.feature` only。
- Do not use a runtime subfolder name that does not match the specs `packages/<current_spec_package>/` directory name。
- Do not infer archive paths outside this runtime instruction file。
