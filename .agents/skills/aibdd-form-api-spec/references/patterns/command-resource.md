# 領域命令、狀態遷移與讀模型

本檔補齊「只寫 REST 形狀」不足以約束的行為面：**誰推進狀態**、**一個業務動作要不要拆 API**、**前端讀什麼**。

## 狀態遷移

- **後端**負責狀態機與規則檢查；OpenAPI 中每個寫入操作應能描述為「在合法前提下，完成一個領域效果並（必要時）推進狀態」。
- **避免**僅為「切狀態」而存在的端點（例如 `/set-status`、`/advance-phase`），除非推理包明示為**稽核／管理後台**且與主流程分離。
- **狀態遷移應作為業務操作之副作用**：例如「提交猜測」同時完成驗證、計算、持久化與回合推進，而非先呼叫「結算」再呼叫「下一回合」兩次無語意保證的 API。

若推理包未寫清狀態機，**回退 Planner 補齊**，不在 contract 隨便發明隱藏狀態。

## 業務邏輯封裝

- **同一使用者意圖**、同一**交易語意**（可對應單一 bounded context 用例）者，**優先**單一 endpoint；避免「一個按鈕對三支 API」卻無資源上的中間狀態可讀。
- **驗證、計算、更新**應在同一操作的伺服器實作內完成；契約上以單一 request／response 表達外觀，不在 OpenAPI 層鼓勵「先 PATCH 再 POST」拼裝。

若一條 .feature Rule 對應多個 HTTP 呼叫，Planner 應回頭檢查是否過度切碎；**預設偏誤**：合併為單一命令。

## 讀模型與輪詢

- **寫入後**客戶端往往需要最新局面：優先在 **200 回應 body** 附**足夠的讀模型**（例如 aggregate snapshot），避免大量 `204` 又无 GET。
- **GET**：對客戶端需要自我同步的資源提供 `GET /{resource}/{id}`（或列表分頁），並在敘述中說明快取／`revision`／ETag 策略若推理包有要求。
- 若需求含**即時多人同步**而契約仍只有 HTTP：以 **GAP** 標記「即時通道未決」（WebSocket、SSE、推播等），不要假設前端只能盲目輪詢。

## 後端職責邊界（契約可觀察的部分）

- **不可把需保密的規則決策下放給客戶端輸入**：例如伺服器才允許知的隨機結果、對手秘密、反作弊關鍵參數，request body **不得**承載「最終裁定結果」除非推理包明示僅測試／示範環境並分 tag 或分 profile。

## 命令型 POST 的資源化寫法（path 無動詞的務實替代）

下列型式擇一，並在推理包保持一致：

- **子集合**：`POST /rooms/{roomId}/members`（加入／開第一席）
- **從屬動作名詞化**：`POST /rooms/{roomId}/game` body 表「開始」或 payload discriminant（若推理包接受）
- **回合／結算**：`POST /rooms/{roomId}/combat/demon-turn` **無 body**，由伺服器依規則結算；**禁止**由客戶端指定違反規則樹的「行動輪盤結果」（正式環境）
