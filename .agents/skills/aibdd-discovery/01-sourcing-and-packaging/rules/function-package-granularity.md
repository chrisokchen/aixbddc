# Function package 顆粒度

### 步驟 B — 用需求敘事做「一條對外主線」檢查

對每個候選包問三個**斬釘截鐵**的問題（全過才保留成獨立一包；任一失敗就**合併**到主線最強的那包，或**拆開**成兩條主線）：

1. **單一對外入口**：是否存在**一個**使用者或第三方可理解的主要入口（例如「完成登入」「送出訂單」），讓這包內的 `.feature` 主軸**不**需要跨包才能完成一條完整 uat slice？
2. **狀態所有權**：這包是否擁有**可講完**的一段狀態生命週期（建立→進行→終止／失敗）而**不**強迫讀者跳到另一包才懂「誰是權威狀態」？
3. **變更爆炸半徑**：若本輪只改這一包，**其餘**候選包是否仍能以「純讀取／純呼叫契約」理解而不必同步改規格？若否，**合併**或重新切主線，直到「改 A 不必連坐改 B」成立。

## 反例

- 用 **技術層名詞** 當包名主軸（`redis-cache`、`utils`）：bottom-up 的單位是 **對外行為切片**，不是基建分層。
- 用 **單一 `.feature` 檔** 當拆包理由：檔是載體；包是**責任邊界**。多個 feature 仍可屬於同一包，只要共用同一條對外主線與狀態所有權。
- **為了對齊人類組織圖** 開包：組織圖是 top-down；這裡只看 impact 表面積與敘事主線。
