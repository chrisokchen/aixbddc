# Step（Action）顆粒度

- **一個 Action**＝**一次**由符合 **Actor（系統用戶）**之觸發者所觸發、且使用者能從介面上感知**完成結果**的**完整業務行動**。
- **觸發者門檻**：Action 必須由**人類使用者**（或等同於使用者主動下達之業務命令，如按鈕、送出表單）**直接觸發**。
- **流程編排／自動推進**（如「全部審核通過後」「批次項目處理完畢後」「條件滿足即進入下一階段」）屬**引擎在前提成立後自動執行**的步驟，**不是** Action；其效果寫在**最後一個使用者 Action** 的 **postcondition**，或以**同一 feature 內多個 Scenario** 表達（含系統後續處理或規則分支），**不**另開 feature file。
- **內建系統處理**（佇列排序、跳過已結案項目、規則引擎分支、彙總計算等）**不得**單獨成 Action；僅在觸發該流程的使用者 Action 或其所屬 feature 規則中描述。
- **Action 名稱**表**業務意圖**，**不得**等同實作細節或內部技術步驟名稱。
- **API-Like Verification**: 你可以用 API 去想像一個 Action 的顆粒度，我們應盡可能確保每一個 Step(Action)/Feature File 都像良好的 API 設計一樣具備底下品質：
    1. 狀態轉換原則
    - [ ] API 是否自行處理所有狀態轉換邏輯？
    - [ ] 是否避免設計專門用於狀態轉換的 API？（如：/set-status 這類 API 不應存在）
    - [ ] 狀態轉換是否作為其他 API 操作的副作用自動完成？
    2. 業務邏輯封裝原則
    - [ ] 相關的業務邏輯是否封裝在同一個 API 中？（避免一個業務動作拆成多個 API）
    - [ ] 驗證、計算、更新等相關操作是否在同一個 API 內完成？
    - [ ] 是否避免出現「流程分解過細」的 API 設計？
    3. 前端職責原則
    - [ ] 前端是否僅負責 UI 呈現與輸入驗證？
    - [ ] 是否避免設計專門用於前端狀態管理的 API？
    - [ ] 是否減少前端對後端的輪詢需求？（如有即時需求，是否考慮 WebSocket 或推播）
    4. 觸發來源原則
    - [ ] 是否必須由使用者主動操作（按鈕／送出）才會發生？
    - [ ] 若需求寫「當…之後自動…」「輪完後系統…」「接著由 X 執行」，是否誤收成獨立 Action？
    針對每個 candidate actions 必須通過上述全部檢核項。
- **Distinct operation**：僅當候選項同時滿足 (a) **可由使用者獨立下達命令觸發**（見觸發者門檻），且 (b) **可獨立驗收其業務結果**，才應對應**另一個** feature file。
  - **僅能命名／僅能描述／僅能在規格段落分節**，不足以拆檔。
  - 同一使用者 Action 內的**多種結果分支**（如多種審核結論、不同計費或折扣結果）→ 用**同一 feature 的多個 Scenario** 覆蓋，**不**因「可獨立命名」而拆檔。

# 反例

- 同一次「業務人員送出客戶評級」意圖，拆成多個 Actions：**送出表單**、**系統計算風險**、**寫入 CRM**、**第三方寄信**——把內部技術鏈當成並列步驟。預期應**收斂為單一 Action**（由該業務人員觸發、名稱表整段業務意圖），風險已計算、CRM 已更新、通知已排程或已寄送等後果，**只**出現在 **postcondition** 或對應 **feature** 之規則敘述，**不**另立系統／第三方 Action。
- 需求：「多位參與者依序提交 → **全部提交完成後**由系統執行後續處置 → 處置有多種結果」——不得拆成 **`提交`** 與 **`系統執行處置`** 兩個 feature。後續處置是**編排自動推進**；多種處置結果是**同一步內分支**，放在最後一個使用者觸發 feature 的 Scenario／規則中，**不**因處置類型可獨立命名而拆檔。
- 把「進入下一工作階段」「建立背景任務」「全部必填完成後送審」拆成 `/enter-next-phase`、`/enqueue-job` 等——同屬**狀態轉換副作用**，應併入最後一個使用者 Action（如 **送出申請**、**完成填寫**），不另立 feature。
- Action 命名成 **`callPaymentGateway`**、**`updateRowInDb`** 等實作或介面名稱，而非可讀的業務意圖。
- **建立訂單**與**取消訂單**兩種可獨立驗收、且可由使用者獨立觸發的 operation，卻共用同一 **binds_feature**／同一顆矩形。
- **下單（會改寫狀態）**綁在僅描述 **查詢訂單列表** 的 feature，或把純 **查詢儀表板**綁在有副作用的付款 feature。
