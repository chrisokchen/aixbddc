# Benchmark 05 — 入職送出與背景查驗

## 埋藏陷阱

- 送出申請後系統**自動**標為審核中、**自動建立**背景查驗任務。
- 查驗通過／需補件／拒絕**三種結果**易各拆一 feature。
- 「進入審核中」「建立背景任務」「呼叫第三方查驗」易成狀態或實作型 feature。

## 需求原文

新人入職：候選人分區塊**填寫入職區塊**（基本資料、證照、銀行帳戶等，可多次儲存）。

當**全部必填區塊皆已完成**後，候選人可**送出入職申請**。送出後系統**自動**將申請標為審核中，並**自動建立**背景身分查驗任務（使用者不可手動建立任務或改狀態）。

查驗完成後有三種結果：通過、需補件、拒絕。HR 可對申請**審核入職申請**（核准或駁回）；若查驗為需補件，候選人補齊後再次送出。

第三方查驗 API、狀態機更新皆為系統內部步驟，不暴露為獨立使用者功能。

## 預期產出之 feature file

- `填寫入職區塊.feature`
- `送出入職申請.feature`
- `審核入職申請.feature`

## 不應出現（常見誤拆）

- `進入審核中.feature`、`建立背景查驗任務.feature`、`執行身分查驗.feature`
- `查驗通過.feature`、`查驗需補件.feature`、`查驗拒絕.feature`
- `設定申請狀態.feature`、`呼叫第三方查驗.feature`

送出後之狀態與查驗任務為 `送出入職申請` 之 **postcondition**；三種查驗結果以 **Scenario** 或審核前置規則表達，不拆檔。
