# Benchmark 04 — 採購問答後進議價

## 埋藏陷阱

- 雙方本輪皆提交後，系統**自動**結束問答、**自動進入議價階段**、**自動計算建議底價**。
- 「進入議價階段」「計算底價」易拆成獨立 feature；底價計算屬**內建系統處理**。
- 議價階段的出價／接受為**新階段下可獨立觸發**之 command，應保留，不與問答混檔。

## 需求原文

採購議價：採購員可對一筆標案**發起議價**，邀請供應商進入問答。

問答階段中，採購員與供應商**輪流**操作：採購員**提交詢問**，供應商**提交答覆**（一輪內各一次）。當**雙方在本輪皆已提交**後，系統**自動**結束本輪問答並**自動進入議價階段**；同時依內部公式**自動計算建議底價**（僅顯示於畫面，無獨立按鈕）。

進入議價階段後，雙方可多次**提交議價出價**；任一方**接受報價**則結案。無使用者操作可單獨觸發「進入議價階段」或「重算底價」。

## 預期產出之 feature file

- `發起議價.feature`
- `提交詢問.feature`
- `提交答覆.feature`
- `提交議價出價.feature`
- `接受報價.feature`

## 不應出現（常見誤拆）

- `進入議價階段.feature`、`系統計算建議底價.feature`、`結束問答輪次.feature`
- `執行議價引擎.feature`、`更新議價狀態.feature`

階段轉換與底價計算併入 `提交答覆`（本輪最後一次使用者提交）之 **postcondition**；出價與接受為議價階段獨立 Action，維持獨立 feature。
