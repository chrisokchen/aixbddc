# Benchmark 01 — 並行會簽

## 埋藏陷阱

- 三位會簽人都提交後，系統**自動**彙總、決議、寫入紀錄、寄通知。
- 核准／退回／補件三種結果與三種通知**可獨立命名**，易誤拆成多個 feature。
- 寫入審批紀錄、寄信屬**內部技術鏈**，易與業務 Action 並列。

## 需求原文

企業內部採購會簽：申請人填寫採購單並**發起會簽**，系統依組織規則指派三位會簽人（可並行）。

每位會簽人登入後可**提交會簽意見**（同意／不同意／需補件），並可附註。申請人在尚未定案前可**撤回會簽申請**。

當**三位會簽人都已提交**後，系統**自動**彙總意見並決議：全數同意則核准；任一人不同意則退回；任一人選需補件且無不同意則標為待補件。決議完成後，系統**自動**寫入審批紀錄，並依結果寄送對應通知（核准／退回／補件各一套範本）。

申請人不可手動「執行彙總」或「觸發通知」；上述為流程引擎在批次完成後的自動步驟。

## 預期產出之 feature file

- `發起會簽申請.feature`
- `提交會簽意見.feature`
- `撤回會簽申請.feature`

## 不應出現（常見誤拆）

- `系統彙總決議.feature`、`執行會簽結論.feature`、`計算會簽結果.feature`
- `寫入審批紀錄.feature`
- `發送核准通知.feature`、`發送退回通知.feature`、`發送補件通知.feature`、`通知申請人.feature`

彙總、寫入紀錄與通知為最後一位會簽人提交後之編排副作用；三種決議結果以 **Scenario** 表達，不另開 feature。
