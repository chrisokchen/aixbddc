# Simplified Wording

把 planner payload（技術語）翻譯成使用者友善的 UI 欄位。目標：不懂內部術語的人也能決策。

---

## 三段式結構

每題呈現必須有三段：

### 1. 目前進度（1 句）

把使用者拉回上下文，讓其記起情境而無需翻上方歷史。

- 模板：`你剛才同意：<事實>` / `我們已經決定：<事實>` / `我們正在處理：<情境>`
- 來源：payload 的 `context` + 會話歷史的前序定案

### 2. 現在的決定（1 句）

問題本身，**不使用任何技術術語**。

- 模板：`現在要決定：<白話問題>`
- 來源：payload 的 `question` 經術語替換

### 3. 選項對照（表格）

好處 / 壞處並列；**禁止直接貼 payload 的 `impact` 原文**。

```
| 選項 | <白話 label> | 好處 | 壞處 |
|------|-------------|------|------|
| A    | ...         | ...  | ...  |
| B    | ...         | ...  | ...  |
| C    | 其他（自行填寫）|   |      |
```

## 術語替換表

| 技術語 | 白話 |
|--------|------|
| top-level boundary | 資料夾 |
| STEP | 步驟 |
| Rule / AtomicRule | 規則 / 單一規則 |
| CiC (BDY/GAP/ASM/CON) | 不提；直接寫問題本身 |
| post-condition / 後置 | 做完後的狀態 |
| pre-condition / 前置 | 做之前的條件 |
| actor / actor key | 角色 |
| authentication prerequisite | 要先登入 |
| idempotency | 同一件事不會做兩次 |
| callback | （付款閘道的）通知 / 通知回傳 |
| subagent | 背景處理 |
| feature file | 功能檔 |
| entity | 資料表 |
| boundary | 模組 / 系統邊界 |
| concurrency | 同時發生的情況 |

有遇到新術語 → 以「若不熟悉此詞的人能看懂」為原則現場改寫。

## 推薦句型

推薦的理由 **用對話語調**，避免結構性描述：

- ✗ 技術：`推薦：B — 語意上屬於購物流程守衛`
- ✓ 白話：`推薦：B — 訪客被擋下來是因為他想買東西，這更像是「結帳流程的守門員」`

- ✗ 技術：`推薦：A — 訂單 entity 有明確 state machine 起點`
- ✓ 白話：`推薦：A — 送出結帳那一刻就有筆訂單，狀態比較好追`

原則：**把因果講出來**，不是抽象名詞的堆疊。

---

## 「好處 / 壞處」推導

clarify-loop 自己從 `option.impact` 推導出成對的好處 / 壞處。推導流程：

1. **拆 impact 句子**：提取動詞動作（新建 / 整合 / 分散 / 簡化 / 增加 / 移除）
2. **判斷方向**：對當前架構或使用者旅程是收斂（好處）、發散（壞處）、還是中性
3. **寫成對照**：`<好處>；<壞處>`；若 impact 明顯只有一面，另一面寫「—」

### 範例

`impact: 需新建 auth/ boundary；訪客觸發登入與未來 login/register 同層`
→
- 好處：未來所有「認證類」feature 集中，好找
- 壞處：要多開資料夾；購物流程會被切散在兩處

`impact: 保持單一 boundary；訪客流程與結帳 feature 同資料夾；未來 auth/ 再建時需決定是否遷移`
→
- 好處：整個購物旅程完整集中
- 壞處：未來若加獨立登入 feature，它們會散落

## 技術定位顯示

題目末尾固定一行：

```
📍 檔案：<location>
```

作為脈絡提示，不影響決策；使用者可忽略。

---

## 反例

### ❌ 技術語殘留
```
[Q1] 「訪客觸發登入」流程歸屬哪個 top-level boundary？
推薦：B — 語意上屬於購物流程的守衛
| A | features/auth/ |
| B | features/checkout/ |
```

### ✓ 白話重寫
```
[Q1]
你剛才同意：訪客按「加入購物車」時要被導到登入頁。
現在要決定：這個「訪客觸發登入」的檔案，要跟誰放一起？
推薦：B — 訪客被擋下來是因為他想買東西，這更像是「結帳流程的守門員」。

| 選項 | 放哪裡 | 好處 | 壞處 |
|------|--------|------|------|
| A | 跟「登入/註冊」一起（auth 資料夾）| 認證類 feature 集中，好找 | 要多開資料夾；購物流程被切散 |
| B | 跟「結帳」一起（checkout 資料夾）| 購物旅程完整集中 | 未來獨立登入 feature 會散落 |
| C | 其他（自行填寫）| | |

📍 檔案：activities/結帳.mmd:19
```
